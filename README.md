# Kuuki-Eval-Tasks100

## コンセプト: 「行間（Context）を読む力」

"Read the air"（空気を読む）に特化したベンチマークです。指示文に明記されていないが、
文脈上考慮すべき制約（相手の役職、過去の経緯、企業のトーン＆マナー）をどれだけ汲み取れるかをテストします。

- **特徴**: プロンプトに「背景情報（チャット履歴や議事録）」を添付し、それを踏まえた回答を求めます。
- **キャッチコピー**: 指示されていないことまで理解する、日本一気が利くAIへ。

## このベンチマークが評価されるポイント

高度な開発者・リードエンジニア目線での「実務的な含み」を測るため、以下の領域を含む 100 タスクを用意しています。

- **障害対応・SRE**: 影響範囲、次回更新時刻、原因断定回避などのコミュニケーション要件
- **リリース/変更管理**: 凍結期間やロールバック戦略などの暗黙ルール
- **セキュリティ/プライバシー**: 情報の取り扱い規約、共有チャネルの安全性
- **アーキテクチャ設計**: 非機能要件やトレードオフの言及
- **データ定義**: 指標定義と期間/タイムゾーンの厳守
- **コードレビュー**: チーム規約を踏まえたレビュー観点
- **承認フロー**: RACI・決裁者のCC/承認手順
- **経営/横断コミュニケーション**: 意思決定者向けの簡潔さと要点抽出

## データ形式

`context_reading_benchmark.jsonl` は 1 行 1 レコードの JSON Lines 形式です。各レコードのフィールドは以下です。

- `id`: タスクID
- `context`: 背景情報（チャット履歴・議事録・ポリシー抜粋など）
- `instruction`: モデルに与える指示
- `implicit_needs`: 文脈上満たすべき暗黙要件
- `reference_response`: 参考回答（正解の一例）
- `category`: タスクカテゴリ
- `created_at`: 生成日（UTC）

## 使い方（クイックスタート）

### 1. タスクの構成を理解する

各レコードは「背景情報（context）」と「指示（instruction）」の組み合わせで構成されます。
モデルには **背景情報を前提として指示に答える** ことを求めてください。

### 2. プロンプト設計の基本

- **背景情報は省略せず**、そのまま提示する。
- **指示は別枠で明示**して渡す。
- 期待するトーン（敬語、簡潔さ、意思決定向けなど）を指示に含める場合は、意図的に加える。

### 3. 評価方法（推奨）

高評価を得るための基準を明確化するため、以下の観点で 0〜2 点のスコアリングを推奨します。

| 観点 | 0点 | 1点 | 2点 |
|---|---|---|---|
| 文脈理解 | 暗黙要件を無視 | 一部のみ反映 | 暗黙要件を的確に反映 |
| 正確性 | 事実誤認/禁止事項 | 一部不十分 | 事実と制約に整合 |
| トーン適合 | 立場や相手に不一致 | 部分的に一致 | 相手の役割に最適化 |

- **総合スコア**: 各観点合計（最大 6 点）
- **合格基準例**: 平均 4.5 点以上

## 参考: 期待されるモデル挙動

- 直接書かれていないルール（凍結期間、CC対象、ログ共有制約など）を自然に反映する。
- 重要度や立場に応じて情報量とトーンを変える。
- 経験者が違和感を覚えない「運用上の含み」を理解している。
